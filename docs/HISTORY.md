# История сообщений

## Что сохраняется

SQLite-хранилище записывает:

- входящие текстовые сообщения пользователя;
- текстовые ответы бота;
- факт отправки презентации, отчёта лидогенерации или выгрузки истории.

Токен Telegram, бинарное содержимое документов и содержимое файлов отчётов не записываются. Для события документа сохраняются только тип, имя и краткое описание результата.

## Область видимости

Каждая запись привязана одновременно к `chat_id` и `user_id`. Запрос истории не может вернуть записи другого пользователя или другого чата. Выгрузка через `/history` разрешена только в личном чате с ботом.

## Срок и объём хранения

- не более 200 последних записей пользователя в одном чате;
- не более 90 дней;
- выгрузка содержит последние 100 записей.

Очистка старых и лишних строк выполняется транзакционно при добавлении нового сообщения.

## Команды

### `/history`

Формирует UTF-8 Markdown-файл `bot-history.md` с датой, временем, направлением и текстом последних сообщений.

### `/clear_history`

Удаляет все записи текущего пользователя в текущем чате. Команда удаления сначала попадает во входящую историю, затем удаляется вместе с остальными строками. Подтверждение не записывается повторно, поэтому после операции список действительно пуст.

## Настройка

Путь задаётся переменной:

```env
HISTORY_DB_PATH=./data/bot-history.sqlite
```

При отсутствии переменной используется значение выше. Родительская папка создаётся автоматически.

## Схема данных

Таблица `message_history` содержит:

| Поле | Назначение |
| --- | --- |
| `id` | Последовательный идентификатор |
| `chat_id` | Telegram chat ID в текстовом представлении |
| `user_id` | Telegram user ID в текстовом представлении |
| `direction` | `user`, `bot` или `system` |
| `text` | Текст до 4000 символов |
| `created_at` | Unix-время в миллисекундах |

Для выборки используется индекс `(chat_id, user_id, created_at DESC)`. SQLite работает в WAL-режиме.

## Эксплуатация и конфиденциальность

- Ограничьте права доступа ОС к каталогу `data`.
- Не публикуйте SQLite-файлы и резервные копии.
- Определите срок хранения резервных копий отдельно: команда `/clear_history` не может удалить внешнюю копию базы.
- Перед промышленным использованием сообщите пользователям о хранении и цели обработки сообщений.
- При нескольких экземплярах бота используйте одно согласованное хранилище или отдельную БД на экземпляр.
